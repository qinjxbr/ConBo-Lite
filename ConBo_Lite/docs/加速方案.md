# ConBo 当前版本加速方案

## 基线与判断原则

`input6.MOV`、两秒、40个抽样帧的旧稳健 Adaptive 流程约需 27.6 秒。新默认 Fast 流程实测 16.06 秒，其中顺序解码并缓存到第240帧为 8.71 秒、搜索为 1.82 秒、全帧精确验证为 5.53 秒；最终仍为240/240完整检出，reference帧仍为46。`input2` 一秒测试为1.22秒，检测中心标准差和跨度均为0 px。

以下建议用三个等级说明对方法的影响：

- **不改变核心原理**：MVGC、圆形 Bokeh、EdgeDrawing、固定 ROI 和相位重建均保留，结果应等价或仅有浮点误差。
- **局部测量改变**：保留整体方法，但替换圆心/位移的局部求解器，输出不保证逐像素一致。
- **重要原理改变**：省略或替换12步中的关键运算，只应作为另一个明确命名的快速模式。

## 已实现：全时段小 ROI 缓存

校准视频只解码一次。每个目标、每帧只保存 `ROI_SIZE x ROI_SIZE` 的 minimum-channel 和 grayscale 数据，不保存完整4K帧。

- 40张4K BGR帧接近1 GB；240个60x60单目标ROI字段只有约1.7 MB。
- 抽样搜索、全帧复核和reference选择共享同一缓存。
- 不改变任何图像运算和评分含义。
- **核心原理影响：无。**

## 已实现：逐级淘汰与条件救援

Fast模式采用 successive halving：

1. 所有粗候选先测试每秒4帧；
2. 前8名使用每秒10帧；
3. 前8名使用全部抽样帧；
4. 约5名抽样决赛者才执行完整圆形卷积和EdgeDrawing；
5. 默认前2名在校准区间全部帧精确验证；若无人100%通过，才触发全时段局部救援。

`input6` 实测由约27.6秒降至16.06秒，约快42%。

- 最终参数仍由原始圆形卷积和EdgeDrawing验证。
- **核心原理影响：无；只改变搜索调度。**

## 已实现：矩特征代理搜索＋精确决赛

搜索早期不为每个组合生成完整圆形响应。代理使用最大连通块占比、占用率、加权质心、协方差特征值、等效半径、边界接触和跨帧稳定性排序。对称圆形卷积不会改变强度质心；代理仅用于安排计算顺序，最终坐标与参数仍由原始圆形卷积和EdgeDrawing决定。

与逐级淘汰结合后的`input6`端到端结果为16.06秒。

- 代理结果不直接成为最终参数和输出，只负责淘汰明显较差组合。
- 最终候选和reference轨迹仍由原算法逐帧验证。
- **核心原理影响：无到极低；最终判据不变。**

## 已实现但不默认：候选×帧批量 GPU

代码可把全部校准ROI常驻CUDA，并缓存gamma校正结果；显式设置`DEVICE="cuda"`即可使用。实测MX450 2 GB在后续连通块仍位于CPU的情况下表现不稳定且可能更慢，因此`auto`的Adaptive代理选择向量化CPU。主流程的大目标批次仍按原阈值自动选择CUDA。

这避免把理论GPU优势误当成当前硬件上的实际收益；更强GPU或更多目标可手动比较JSON中的分段耗时。

- 数学操作可保持一致。
- **核心原理影响：无；只有浮点舍入差异。**

## 已实现接口：OpenCV/FFmpeg 硬件解码与编码

`IO_ACCELERATION`支持`cpu/auto/hardware`。当前OpenCV构建会尝试硬件后回退并可能打印FFmpeg警告，因此安静可靠的默认是`cpu`；具备正确硬件编解码构建时可显式切换。系统PATH仍没有独立FFmpeg，代码不会擅自安装外部依赖。

- `input6` 两秒解码约8.8秒，预计可降至约3–5秒。
- 与方案一、二结合，Adaptive预计约7–12秒。
- 完整4K Track/MAG输出预计提升1.5–3倍，具体取决于编码器和磁盘。
- **核心原理影响：无；编码视频可能有不同的有损压缩误差。**

## 明确不采用：Track 矩中心＋周期复核

按用户决定，不实现该方案。Track每个目标、每一帧都完整执行MVGC、圆形Bokeh和EdgeDrawing；代码和CSV均显式记录`full_circular_edgedrawing_every_frame`。

- 局部中心求解实测约2.5倍加速。
- 4K端到端仍受解码/编码限制，预计整体提升10%–40%。
- 圆形Bokeh不再是每一帧唯一的中心测量器，而成为初始化和复核器。
- **核心原理影响：中等。Tracking 的局部检测步骤发生改变，应作为 `TRACK_DETECTOR="hybrid_moment"` 可选模式，不能静默替换 exact 默认。**

## 已实现可选模式：MAG phase correlation

固定ROI天然适合相位相关。快速模式在Gaussian响应上直接求 reference/current 的亚像素位移；原始帧不变形，当前圆的半径、轴和角度仍由圆形Bokeh实际检测，只给这个已检测圆追加 `(amplification-1) * phase_shift`。

- 实测相位局部运算提升约9.5–12.8倍；4K短片端到端提升约7%，因为视频写入占主导。
- 仍使用固定ROI和频域相位，但不再执行“放大相位后重建图像”的原始步骤。
- **核心原理影响：中到高。它测量并数值放大位移，而不是重建放大Bokeh，应命名为 `MAG_ALGORITHM="phase_shift"`；现有 `phase_reconstruct` 必须保留为论文/方法一致模式。**

## 已实现：输出层加速

`WRITE_VIDEO=False`只写CSV；`OUTPUT_SCALE=0.5`让分析保持原分辨率但输出半尺寸视频。

- 只分析CSV预计可提升2–5倍。
- 半尺寸视频的像素量为四分之一，编码/绘制常可提升2–3.5倍。
- 分析坐标和算法不变，但可视化文件分辨率改变。
- **核心原理影响：无。**

## 推荐实施顺序

当前前四项均已落地；Track hybrid明确不实现。MAG继续以`phase_reconstruct`为默认，并保留已实现的`phase_shift`可选模式。

前四项不会改变方法的基础重要原理。第五项改变Tracking的逐帧局部检测器；第六项改变MAG的放大实现，应分别作为显式可选接口而不是替换论文一致模式。
